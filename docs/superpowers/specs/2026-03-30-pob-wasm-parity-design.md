# pob-wasm Full Parity Design Spec

**Date:** 2026-03-30
**Status:** Draft
**Predecessor:** `2026-03-29-pob-wasm-design.md`
**Scope:** PoE 1 (3.x) only. PoE 2 deferred.

## 1. Problem Statement

The initial implementation covers roughly 5-8% of Path of Building's calculation engine by functional coverage. The architecture (crate layout, ModDb, CalcEnv pipeline, WASM wrapper, CI) is sound, but the calculation logic is skeletal:

| System | Current State | PoB Lua Reference |
|--------|--------------|-------------------|
| Item processing | Missing entirely | CalcSetup.lua ~1,400 lines |
| ModParser (stat text to mods) | 25 patterns | ModParser.lua 6,762 lines (~5,650 patterns) |
| Offence calc | 195 lines, simplified | CalcOffence.lua 5,953 lines |
| Defence calc | 55 lines | CalcDefence.lua 3,825 lines |
| Perform (life/mana/regen/leech) | 100 lines | CalcPerform.lua 3,636 lines |
| CalcSetup (env init) | 85 lines | CalcSetup.lua 1,789 lines |
| Triggers | 100 lines, wrong formulas | CalcTriggers.lua 1,571 lines |
| Active skill resolution | 110 lines, heuristic | CalcActiveSkill.lua 902 lines |
| ModStore/EvalMod (tag evaluation) | Not implemented | ModStore.lua 903 lines (hot path) |
| Data files | Structural stubs | 667K lines of Lua data tables |

Structural problems in the current codebase:

- **Oracle tests are broken by design.** `assert_output_approx` silently passes when keys are missing. Several tests assert on keys absent from expected data.
- **Test builds are trivial.** Single gem, 0-1 passives, no items, no ascendancy. They exercise only the basic flat stat path.
- **Expected values appear hand-written**, not generated by `gen_oracle.lua` against actual PoB.
- **`is_support` is hardcoded to `false`.** Support gems are never identified or processed.
- **No `skillCfg`** is ever constructed. All modifier queries ignore flag/keyword filtering.
- **No `EvalMod`** tag evaluation. Conditional mods, per-stat scaling, multipliers are silently ignored.

## 2. Goal

**Literal 100% parity** with the Path of Building Community Fork's PoE 1 calculation engine.

What this means concretely:

- 30+ real-world oracle builds covering all archetypes (full passive trees, items, support gems, ascendancy, jewels, flasks)
- ALL fields in `env.player.output` are compared per oracle build (not 1-2 cherry-picked fields)
- `assert_output_approx` in strict mode: fails if expected has a key that actual lacks (and vice versa for a configurable set of required keys)
- Numeric tolerance: 0.1% for all values
- Breakdowns populated and structurally matching PoB's format
- Special-case unique items included in oracle builds
- Timeless jewels, cluster jewels, jewel radius effects functional
- Every codepath in CalcSetup, CalcPerform, CalcDefence, CalcOffence, CalcActiveSkill, CalcTriggers, CalcMirages, and ModParser ported

**Out of scope:** PoE 2 mechanics, UI/display concerns, `getCalculator()` incremental mode, `CalcSections.lua` (UI layout definitions).

## 3. Approach

**Foundation first, then full calc modules.**

The current implementation's core problem is that the foundation is missing. ModParser handles 25 of ~5,650 patterns, items are not processed, support gems are ignored, EvalMod does not evaluate mod tags. Without this foundation, every calc module is forced to fake its inputs.

Building the foundation (ModStore/EvalMod, ModParser code-gen, data pipeline, item/support processing) first means every subsequent calc module can be ported faithfully from the Lua, line-by-line, with confidence that mods flow correctly into ModDb.

## 4. Phase Structure

```
Phase 1:  Oracle Infrastructure Rebuild
    |
Phase 2:  ModStore/EvalMod + Mod Type System
    |
Phase 3:  ModParser Code Generator
    |
Phase 4:  Data Pipeline (full fidelity from GGPK)
    |
Phase 5:  Build Processing (items, supports, flasks, jewels, CalcSetup full port)
    |
Phase 6:  CalcPerform Full Port
    |
Phase 7:  CalcDefence Full Port
    |
Phase 8:  CalcOffence Full Port
    |
Phase 9:  CalcTriggers + CalcMirages + Calcs.lua Full Port
    |
Phase 10: Long Tail (special cases, edge cases, all 30+ oracle builds green)
```

Phases 1-5 are the foundation and must be completed in order. Phases 6-8 operate on different output fields and could technically parallelize, but all depend on Phase 5. Phase 9 depends on Phases 6-8. Phase 10 is iterative convergence.

**Each phase's exit criteria:** all existing oracle tests still pass + new oracle builds added for that phase's systems pass within 0.1% tolerance.

## 6. Phase 2 — ModStore/EvalMod + Mod Type System

### Problem
The current ModDb has `sum/more/flag/tabulate` but no mod tag evaluation. PoB's `EvalMod` (ModStore.lua, 592 lines) evaluates ~20 tag types on every modifier query. Without it, conditional mods, per-stat scaling, multipliers, and skill-filtered queries return wrong values.

### Deliverables

**2.1 Expand `ModFlags`**
Add all weapon-type flags:
```
WAND, CLAW, SWORD, AXE, MACE, STAFF, DAGGER, SCEPTRE, BOW,
SHIELD, DUAL_WIELD, TWO_HAND, ONE_HAND, DOT_MULTI, CAST,
BRAND, CHANNELLING
```

**2.2 Expand `KeywordFlags`**
Add all keyword flags:
```
AURA, CURSE, WARCRY, HEX, MARK, BRAND, TOTEM, TRAP, MINE,
MINION, CHANNELLING, HERALD, VAAL, INSTANT, MOVEMENT, TRAVEL,
LIGHTNING, COLD, FIRE, CHAOS, PHYSICAL
```

**2.3 Implement `ModTag` enum**
Port all tag types from ModStore.lua's `EvalMod`:

| Tag | Behavior |
|-----|----------|
| `Multiplier { var }` | Value multiplied by `modDB.multipliers[var]` |
| `MultiplierThreshold { var, threshold }` | Active only if multiplier >= threshold |
| `PerStat { stat, div }` | Value scaled by `output[stat] / div` |
| `PercentStat { stat, percent }` | Value scaled by `output[stat] * percent / 100` |
| `StatThreshold { stat, threshold }` | Active only if `output[stat] >= threshold` |
| `DistanceRamp { ramp: Vec<(f64, f64)> }` | Distance-based interpolation |
| `Limit { var }` | Caps total contribution of this mod name |
| `Condition { var, neg }` | Boolean condition check against `modDB.conditions` |
| `ActorCondition { actor, var, neg }` | Condition on a specific actor's ModDb |
| `ItemCondition { slot, var }` | Condition on an equipped item property |
| `SocketedIn { slot }` | Mod only applies to gems socketed in this slot |
| `SkillName { name }` | Mod only applies to the named skill |
| `SkillId { id }` | Mod only applies to the skill with this ID |
| `SkillPart { part }` | Mod only applies to a specific skill part |
| `SkillType { type_flags }` | Mod only applies if skill has matching type flags |
| `SlotName { slot }` | Mod scoped to items in this slot |
| `ModFlagOr { flags }` | Alternative flag matching (any flag matches) |
| `KeywordFlagAnd { flags }` | Required keyword matching (all flags required) |
| `GlobalEffect { effect_type, unscalable }` | Buff/debuff categorization |

**2.4 Implement `eval_mod()`**
Port ModStore.lua's `EvalMod` function. Signature: `fn eval_mod(mod_entry: &Mod, cfg: &SkillCfg, mod_db: &ModDb, output: &OutputTable) -> Option<f64>`. Returns `Some(scaled_value)` if the mod applies, `None` if it does not.

**2.5 Implement `SkillCfg` type**
```rust
pub struct SkillCfg {
    pub flags: ModFlags,
    pub keyword_flags: KeywordFlags,
    pub slot_name: Option<String>,
    pub skill_name: Option<String>,
    pub skill_id: Option<String>,
    pub skill_part: Option<u32>,
    pub skill_types: u64,
    pub skill_cond: HashMap<String, bool>,
    pub skill_dist: Option<f64>,
}
```
This is passed to every `modDB:Sum/More/Flag` call in offence/defence.

**2.6 Refactor `ModDb.sum/more/flag/tabulate`**
Integrate `eval_mod()`. All queries accept an optional `&SkillCfg` and `&OutputTable`. Current queries that pass `None` behave as before (no filtering). Queries with a cfg use `eval_mod` to filter and scale each mod.

**2.7 Implement `Override` and `Max` handling**
In `sum()`: if any matching mod has `ModType::Override`, return that value (last override wins). If any has `ModType::Max`, the sum is capped at the Max value.

**2.8 Implement `List` type queries**
`mod_db.list(cfg, name) -> Vec<&ModValue>` returns all List-type mod values matching the name. Used for complex effects like added skill mods, enemy modifier lists, etc.

**2.9 Parent chain with cfg forwarding**
Ensure parent ModDb queries forward the same cfg and output references.

### Lua Reference
- `third-party/PathOfBuilding/src/Classes/ModStore.lua` lines 312-903 (`EvalMod`)
- `third-party/PathOfBuilding/src/Classes/ModDB.lua` lines 1-357
- `third-party/PathOfBuilding/src/Classes/ModList.lua` lines 1-245

### Exit criteria
Unit tests demonstrate `eval_mod` correctly processes all 20 tag types. `ModDb.sum()` with a `SkillCfg` returns different values than without. Existing oracle tests still pass.

## 7. Phase 3 — ModParser Code Generator

### Problem
`item_parser.rs` handles 25 stat text patterns. ModParser.lua has ~5,650 patterns across `modNameList` (~1,827 lines) and `specialModList` (~3,820 lines). Without these, items, passives, jewels, and enchants contribute almost no stats.

### Deliverables

**3.1 Build-time Lua parser tool** (`tools/modparser-codegen/`)
A Rust binary that:
- Reads `third-party/PathOfBuilding/src/Modules/ModParser.lua`
- Parses `formList` (regex patterns for INC/RED/MORE/LESS/BASE/FLAG forms)
- Parses `modNameList` (~1,827 lines of pattern-to-modifier mappings)
- Parses `specialModList` (~3,820 lines of pattern-to-function-to-modifier mappings)
- Emits generated Rust source (`mod_parser_generated.rs`) containing:
  - `FormPattern` enum with all form variants
  - `parse_mod_name(line: &str) -> Option<Vec<Mod>>` with match arms for modNameList
  - `parse_special_mod(line: &str) -> Option<Vec<Mod>>` with match arms for specialModList
  - `parse_mod(line: &str) -> Vec<Mod>` pipeline matching ModParser.lua's `parseMod()` flow

**3.2 Lua pattern to Rust regex translation**
Lua patterns (`%d+`, `(%a+)`, `([%a ]+)`) translated to Rust regex equivalents. Captures extracted and used in mod construction. Handle Lua-specific pattern features: `%d`, `%a`, `%w`, `%l`, `%u`, `%s`, `%p`, character classes, anchors.

**3.3 Mod construction from pattern matches**
Each pattern produces structured `Mod` values with proper `ModType`, `ModFlags`, `KeywordFlags`, `Vec<ModTag>`, and `ModValue`. The specialModList entries are Lua functions — the code generator emits Rust functions performing the same dynamic construction based on regex captures.

**3.4 `build.rs` integration**
Code generator runs at `cargo build` time if `ModParser.lua` has changed (file content hash check). Generated file placed in `OUT_DIR` and `include!`'d into the crate.

**3.5 Verification**
Unit tests comparing parse output for 200+ representative stat lines against PoB's Lua output. A Lua test script runs `ModParser.parseMod(line)` on the same inputs and serializes results for comparison. Coverage must include:
- All forms (increased, reduced, more, less, base, flag, override)
- Weapon-specific mods (local and global)
- Conditional mods ("while on Full Life")
- Gem-specific mods ("Supported Skills deal N% more Damage")
- Multi-stat mods ("Adds N-M Fire Damage to Attacks and Spells")
- Resistance mods, attribute mods, life/mana/ES mods
- Critical strike mods, leech mods, penetration mods
- At least 10 specialModList patterns with dynamic captures

**3.6 Replace `item_parser.rs`**
The hand-written parser is replaced entirely by the generated code.

### Lua Reference
- `third-party/PathOfBuilding/src/Modules/ModParser.lua` (entire file, 6,762 lines)

### Exit criteria
`parse_mod("10% increased maximum Life")` returns correct `Mod { name: "Life", mod_type: Inc, value: 10.0, ... }`. Same for 200+ diverse stat lines. Generated code compiles. `item_parser.rs` replaced.

## 8. Phase 4 — Data Pipeline (Full Fidelity from GGPK)

### Problem
Data files are stubs. `gems.json` has 4 hand-crafted entries. `bases.json` has no actual stats. `mods.json` has no stat values. `tree/poe1_current.json` has 8 nodes with stats.

### Deliverables

**4.1 Gems — full extraction**

Per-level stats: base damage (min/max per element), crit chance, cast time, attack speed multiplier, mana cost, damage effectiveness, cooldown, duration, stored uses.

Gem metadata: `is_support`, `skill_types` (all flags from `SkillType` enum — 40+ types), gem tags, quality stats (normal/anomalous/divergent/phantasmal), support gem matching rules.

Support gem data: the modifiers a support gem adds to supported skills (e.g., "Supported Skills deal 30% more Physical Damage"). Matching expression (which skill types the support can support). Cost multiplier.

Output: `data/gems.json` — at least 500 gems with level data, all supports with their modifier lists.

**4.2 Base item data — full extraction**

- Weapon bases: physical damage range (min/max), attack speed, crit chance, range, weapon class, weapon type
- Armour bases: base armour, base evasion, base ES, base ward
- Shield bases: base block chance
- Flask bases: life/mana recovery, duration, charges used/max
- Jewel bases: type (regular, abyss, cluster, timeless)
- Implicit mods per base type
- Requirements (str/dex/int/level)

Output: `data/bases.json` — at least 200 weapon bases with damage stats, all armour/shield bases with defensive stats.

**4.3 Mod data — full extraction**

- Stat values: stat_id, min value, max value per stat per mod tier
- Mod domains: item, flask, jewel, crafted, veiled, enchant, etc.
- Mod groups for mutual exclusion
- Mod weight/tags for generation context

Output: `data/mods.json` — at least 10,000 mods with stat values.

**4.4 Passive tree — full extraction**

- ALL nodes with stat descriptions (5,500+ nodes)
- Node types: small, notable, keystone, mastery, jewel_socket, ascendancy, class_start
- Ascendancy nodes with their ascendancy class
- Mastery effects (separate from node stats, selectable per mastery)
- Jewel socket positions (orbit, orbit_index) for radius calculations
- Node group positions for Timeless Jewel transformations
- Cluster jewel notables

Output: `data/tree/poe1_current.json` — all nodes with stats and types.

**4.5 Unique item data**

Extract or generate `data/uniques.json` with unique items and their fixed mods. At minimum, all uniques referenced in CalcSetup.lua's special-case handling.

**4.6 Data validation tests**

A test that loads all JSON files and verifies:
- Gems: >= 500 gems with level data, >= 200 supports identified
- Bases: >= 200 weapon bases with damage stats
- Tree: >= 1,000 nodes with stats, >= 50 keystones, >= 100 notables per ascendancy grouping
- Mods: >= 10,000 mods with stat values

**4.7 Update `GameData`**

Expand the struct to hold: `gems`, `bases`, `mods`, `uniques`, `misc`, `passive_tree`. Update `from_json()` to load all files. Add `bases.rs` data types (currently empty stub).

### Exit criteria
Data files populated with real GGPK-extracted data. `GameData::from_json()` loads all files. Validation tests pass.

## 9. Phase 5 — Build Processing (CalcSetup Full Port)

### Problem
`setup.rs` is 85 lines. CalcSetup.lua is 1,789 lines. Items, support gems, flasks, jewels, ascendancy, and special-case items are completely absent.

### Deliverables

**5.1 Item XML parsing**
Parse `<Item>` elements from build XML (currently skipped by `xml_parser.rs`). Parse item text blocks into structured `Item` data: rarity, base type, implicits, explicits, crafted mods, enchants, corrupted implicits, influence mods, quality, sockets, links. Use generated `parse_mod()` from Phase 3 to convert each mod line to `Vec<Mod>`. Apply item quality scaling (armour/evasion/ES quality, weapon quality).

**5.2 Item slot processing**
Port `initEnv()` item loop. Create per-slot ModDb layers (Helmet, BodyArmour, Gloves, Boots, Weapon1, Weapon2, Ring1, Ring2, Amulet, Belt). Add item mods to slot ModDb with `ModSource` attribution. Handle local vs global mod distinction. Extract weapon data (base damage, attack speed, crit) to `env.player.weaponData1/weaponData2`. Extract shield data (base block). Detect dual-wield, two-hand, shield-equipped.

**5.3 Support gem processing**
Port `calcs.buildActiveSkillModList()` from CalcActiveSkill.lua (~619 lines). Fix `is_support: false` bug. Match supports to active skills via `canGrantedEffectSupportActiveSkill()`. Apply support gem mods to the active skill's ModDb. Build `supportList` on each active skill. Handle cost multipliers, added damage, flag additions.

**5.4 Active skill construction**
Port `calcs.createActiveSkill()`. Build skill flags from gem's `skillTypes` (data-driven, replacing hardcoded name lists). Build `skillCfg` with correct flags, keywordFlags, slotName, skillName, skillPart. Construct damage conversion table.

**5.5 Flask processing**
Parse flask items. Apply flask mods when configured as active. Handle flask effect modifiers.

**5.6 Jewel processing**
- Regular jewels: mods added to actor ModDb
- Abyss jewels: socket into items, add mods with slot scoping
- Cluster jewels: sub-tree with notables
- Timeless jewels: transformation of tree nodes within radius
- Jewel radius calculation from socket position

**5.7 Buff/aura/curse setup**
Process aura skills as buffs. Process curse skills as enemy debuffs. Store in `env.buffList` / `env.curseList`.

**5.8 Ascendancy processing**
Ascendancy nodes identified by type. Ascendancy notables with special mechanics handled per-class where needed.

**5.9 Special-case uniques**
Port ALL special-case item handling from CalcSetup.lua's `initEnv()`:
- Energy Blade, Kalandra's Touch, The Iron Mass, Dancing Dervish, Widowhail, Necromantic Aegis
- All other special cases

**5.10 `initModDB()` full port**
Per-class base stat lookup tables (not hardcoded formulas). Base accuracy formula. All default conditions and multipliers from CalcSetup.lua's `initModDB()`.

**5.11 Enemy setup**
Initialize `env.enemy.modDB` with enemy level-based stats from monster tables. Base enemy resistances. Curse effectiveness base.

### Lua Reference
- `third-party/PathOfBuilding/src/Modules/CalcSetup.lua` (1,789 lines)
- `third-party/PathOfBuilding/src/Modules/CalcActiveSkill.lua` lines 82-163, 227-845

### Exit criteria
A real-world build with full tree, items, support gems, and ascendancy produces a populated ModDb with thousands of correctly categorized mods. Oracle tests on "physical melee with gear" and "spell caster with supports" pass for basic stats (Life, Mana, attributes) within 0.1%.

## 11. Phase 7 — CalcDefence Full Port

### Problem
`defence.rs` is 55 lines. CalcDefence.lua is 3,825 lines.

### Deliverables

**7.1 Resistance calculation** — `calcs.resistances()` (L509-635)
Per-element (Fire/Cold/Lightning/Chaos). Base + uncapped. Max resist (75% + mods). Capping. OverCap tracking. Damage-taken-as conversion effect. Melding of the Flesh. Enemy penetration.

**7.2 Armour/Evasion/ES from gear**
Per-slot: local base + local inc/more -> slot total. Sum across slots. Global inc/more on total. Iron Reflexes (Evasion->Armour). Unwavering Stance. Grace/Determination/Discipline flat additions.

**7.3 Block**
Attack block (shield base + mods, cap 75%). Spell block (base + mods, cap 75%). Staff block. Dual wield bonus. Glancing Blows (cap 75%, only prevents 65%).

**7.4 Spell Suppression**
Suppression chance (Dex + mods). Effect (50% less spell damage). Cap 100%.

**7.5 Hit avoidance**
Evade chance formula. Avoid elemental ailments. Avoid stun. Avoid poison/bleed/impale.

**7.6 Physical damage reduction** — armour formula
`reduction = armour / (armour + 5 * damage)`. Endurance charge contribution. Cap at 90%. Additional sources.

**7.7 Damage taken conversion** — `calcs.applyDmgTakenConversion()`
"X% of Physical Damage taken as {Element}". Applied before resistance.

**7.8 Pool reduction cascade** — `calcs.reducePoolsByDamage()` (L163-409, 247 lines)
Full PoE order: Frost Shield/allies -> Aegis -> Guard skill -> Ward -> ES (with CI/bypass rules) -> MoM (mana absorption) -> Life loss prevention -> Life -> Overkill.

## 15. Testing Strategy

### Three-layer testing (expanded from original spec)

**Layer 1: Unit tests (native, fast)**
Inline `#[test]` in every module. Construct ModDb directly, test individual functions. These are fast and run on every build.

**Layer 2: Oracle parity tests (against PoB ground truth)**
30+ builds in `crates/pob-calc/tests/oracle/`, each with `.xml` and `.expected.json`. Tests run with `DATA_DIR` pointing to extracted game data. `assert_output_approx` in strict mode compares ALL output fields within 0.1% tolerance.

**Layer 3: WASM integration tests (browser)**
`wasm-bindgen-test` in headless Chrome verifying the full JS API pipeline.

### Oracle test fix requirements
1. `assert_output_approx` must fail on missing keys (not silently pass)
2. Tests must NOT silently skip — use `#[ignore]` attribute with clear reason, or require DATA_DIR in CI
3. Expected JSON generated by `gen_oracle.lua` against actual PoB (not hand-written)
4. Each oracle test compares ALL numeric output fields (not 1-2)
5. Oracle CI in Docker with LuaJIT + PoB submodule

## 17. Architectural Decisions

### ModDb ownership model
`CalcEnv` owns `player: Actor` and `enemy: Actor`. Each Actor owns its `mod_db: ModDb`. Per-slot ModDb instances hold local item mods and have the actor's ModDb as their parent (for `sum/more/flag` queries that need to include global mods). When calculating slot-specific stats (e.g., local weapon DPS), queries run against the slot ModDb. When calculating actor-level stats (e.g., total Life), queries run against the actor ModDb which aggregates global mods from all items. This mirrors PoB's `ModStore` parent-chain architecture.

### SkillCfg threading
Every `mod_db.sum/more/flag/tabulate` call in offence/defence passes the active skill's `SkillCfg`. Calls outside of skill-specific contexts (e.g., life calculation) pass `None`. This is the most critical architectural change — it makes mod tag evaluation work.

### Generated code strategy
The ModParser code generator produces a single `.rs` file with static match tables. This is `include!`'d at compile time. The generator is a separate Rust binary in `tools/modparser-codegen/`.

## 18. Risk Assessment

| Risk | Mitigation |
|------|-----------|
| ModParser code generator may not handle all specialModList Lua functions | Start with modNameList (simpler), add specialModList incrementally. Verification against 200+ stat lines catches gaps early. |
| GGPK extraction may not produce all required data fields | Fall back to PoB's Data/*.lua files for specific missing fields. The data-extractor can be extended. |
| CalcOffence.lua's 5,634-line function is error-prone to port | Port section by section (weapon damage, then conversion, then crit, etc.). Each section gets its own oracle build validation. |
| EvalMod performance on WASM | Profile after Phase 2. If hot-path is slow, optimize with pre-computed flag masks. |
| Timeless jewel seed-based transformation is complex | Defer to Phase 10 long tail. Most builds don't use timeless jewels. |
| Some PoB Lua uses metatables and dynamic dispatch | Rust enums and trait objects serve the same purpose. Map idioms carefully. |
